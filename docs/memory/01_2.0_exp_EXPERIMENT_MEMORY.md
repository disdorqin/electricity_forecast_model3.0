## 2.0_exp 做过什么

2.0_exp 曾构建过多模型预测 + 动态权重学习 + 融合 + 负电价修正链路，核心实验链路为：

```text
R3D-Tap-GEF Pipeline
```

旧链路阶段：

```text
Stage 1: validation tap + real forecast
Stage 2: learner 学习权重
Stage 3: fusion 加权融合
Stage 4: negative price classifier + final outputs
```

## 关键历史经验

### business_day 规则

所有模型输出、fold 切分、真实值 merge、融合、final 输出必须使用：

```text
business_day + hour_business
```

不能只用：

```text
ds.date()
ds.normalize()
```

关键规则：

```text
hour_business = 24 of business_day D => ds = D+1 00:00:00
```

标准字段：

```text
business_day
ds
hour_business = 1..24
period = 1_8 / 9_16 / 17_24
```

### y_true 覆盖率修复经验

不能依赖模型输出自带 y_true。应该统一从原始数据按：

```text
task + business_day + hour_business
```

补齐 y_true。

### `--models all` 修复经验

`--models all` 或不传 `--models` 时，应该展开为正式模型列表，不能把 `"all"` 当模型名。

### loader 经验

需要统一 `load_table()`：

```text
支持 .csv / .xlsx / .xls
支持 utf-8 / utf-8-sig / gbk / gb18030
支持中文列名映射
```

### TimesFM / TF 经验

主链路只能有一个 TimesFM 入口。TF 或 legacy 文件只能进入 archive，不允许 production pipeline 多处搜索入口。

### GPU OOM 经验

DA + RT 连续跑时需要资源隔离。优先方案：

```text
split-target subprocess
每个 target 单独 Python 进程
```

必要时：

```text
gc.collect()
torch.cuda.empty_cache()
```

## 3.0 继承原则

3.0 不继续修补 validation tap，而是保留其经验：

- business_day / hour_business 口径
- y_true 回填与对齐策略
- loader 稳定性
- GPU 资源释放经验
- negative classifier 接口经验
- fusion learner 的 metric 经验

## 3.0 不继承

- validation tap 作为生产主线
- rolling OOF 作为核心权重学习机制
- 每日重跑 D-30~D-1 来学习权重
- 多入口 TimesFM / TF 搜索
- 臃肿 scripts/runners/services 混杂结构
